Appendix B: Tracker Tag Firmware 37 Abstract Outdoor cats are exposed to risks such as becoming lost, entering unsafe areas, or experiencing delayed recovery due to lack of real-time location visibility. Conventional tracking solutions often rely on cellular infrastructure, resulting in limited coverage, increased operational costs, and reduced reliability in rural or obstructed environments. This project addresses these limitations through the design and implementation of a low-power, long-range Outdoor Cat Tracker system based on LoRa communication. The system consists of both hardware and software subsystems operating as an integrated platform. The hardware subsystem includes a vest-mounted tracking device composed of a GPS module, LoRa transceiver, microcontroller, and rechargeable battery, designed to operate under strict size, weight, and power constraints. Location data is periodically acquired and transmitted via LoRa to a home-base receiver, which forwards the data to a backend server using local internet connectivity. The software subsystem includes a cloud-integrated backend and a cross-platform mobile application developed using React Native and Firebase services. The application provides real-time map-based tracking, geofence configuration, device management, and event-based alerting. A structured data model ensures consistency across users, pets, devices, and alert records, while geofence logic enables detection of safe-zone exit and return conditions. The implemented system demonstrates reliable end-to-end operation, including data acquisition, wireless transmission, cloud synchronization, and user-facing visualization. The results validate the effectiveness of LoRa-based communication for localized pet tracking applications, providing a scalable and cost-efficient alternative to traditional cellular solutions.

Introduction & Background The purpose of this project is to design and implement a complete geolocation tracking system for outdoor pets, where the pet’s location is continuously available to the owner through a mobile interface. The system requires a physical tracking device attached to the pet, a communication mechanism to transfer location data, and a backend infrastructure to store and serve this data to the user. Tracking an outdoor cat introduces several engineering constraints. The device must be lightweight and compact to avoid interfering with the animal, while maintaining sufficient battery capacity for extended operation. Additionally, the system must provide location accuracy within an acceptable range (approximately 5–10 meters offset ) and ensure reliable communication even in environments where cellular coverage is weak or unavailable. To address these constraints, LoRa communication was selected as the primary data transmission method. Unlike cellular-based solutions, LoRa provides long-range, low-power communication, making it suitable for outdoor environments and rural deployment scenarios [1]. In this system, GPS data collected by the tracker is transmitted to a home-base receiver, which then forwards the data to a backend server via a local internet connection. The backend server is responsible for processing, storing, and serving location data to the mobile application. The application must accurately display the pet’s position on a map, support user authentication, and allow configuration of tracking parameters such as geofence boundaries. Additionally, the system must support multiple devices per user, requiring a structured method for device identification and assignment. The final implemented system successfully integrates embedded hardware, wireless communication, cloud services, and mobile software into a unified platform. It enables real-time tracking, geofence-based monitoring, and alert generation, providing a practical solution for improving outdoor pet safety.

Objectives The primary objective of this project is to design and implement a reliable, low weight and power Outdoor Cat Tracker system that integrates embedded hardware, wireless communication, cloud infrastructure, and a mobile application to provide real-time pet monitoring. The first objective is the development of the hardware tracking device, which must be compact, lightweight, and energy-efficient. The device is required to accurately acquire GPS location data and transmit it periodically using a LoRa communication module while maintaining acceptable battery life. The second objective is to establish a robust communication pipeline that enables reliable data transfer from the tracker to the backend server. This includes implementing a LoRa-based transmission system and a home-base receiver that forwards data to the cloud using standard internet connectivity. The third objective is the design and implementation of a cloud-based backend system capable of storing and managing user data, pet profiles, device ownership, geofence configurations, and alert history. The backend must ensure data consistency and support real-time updates for the mobile application. The fourth objective is to develop a cross-platform mobile application that provides real-time visualization of pet location, user authentication, and system interaction. The application must present location data clearly through a map interface and allow users to manage pets, devices, and tracking settings. The fifth objective is to implement geofence-based monitoring, where the system evaluates the pet’s position relative to a defined safe zone and detects transition events such as exit and return. These events must trigger alerts and be recorded in the system for user reference. The final objective is system integration and validation, ensuring that all subsystems operate together reliably. This includes verifying end-to-end data flow from the tracker to the mobile application, validating geofence event detection, and confirming correct system behavior under typical operating conditions.

3 - Ceramic Patch Antenna Due to design constraints a long whip-type antenna would not be viable for this project. Various other types of antennas were researched before deciding on a 25x30x5mm Ceramic patch antenna. It has a gain of +2.0dbi which is significantly less than a long whip-type antenna (+10dbi) but still greater than a tiny ceramic “chip-type” antenna with only +0.1dbi. In addition it connects using the RAK3172 breakout boards built-in SMA port. Attached are graphs describing the relationship between distance and Received Strength of Signal Integrity (RSSI) using the two different antenna types and also using two different transmission power modes. A lower power mode with transmission power set to +10db and a higher power mode with it set to +20db. These are based on theoretical estimates of exponential signal decay. Figure 1.: RSSI over distance when TX outputs (a) 10dBm , and (b) 20dBm of power rssi = (rssi_1m_ref + gain) - 10 * n * np.log10(d / d0) Signal strength was modeled to have logarithmic decay. A factor n = 2.5 was chosen to model in all cases. With further testing and data collection this factor can be refined. The rssi_1m_ref was measured to be -35 during low power tx mode and -25 during high power with the whip-type antenna. The expected minimum range using low-power and the patch antenna is between 350-900 meters depending on the environment. This is well within the normal housecats territory range. 4 - Power Solution To power the transceiver, MCU and GPS sensor a power solution consisting of 3 main components was designed. TC4046 Charging Circuit takes in a 5V input using USB-C and charges a LI-PO battery at 1C. It also contains indicator LEDS to show when fully charged or when actively charging. The battery chosen is a 3.7V 600mah 603040 Li-Po which is capable of handling the full current draw of the circuit. The RAK3172 has a max operating voltage of 3.6V and so a LDO was needed to regulate the batteries voltage down to 3.3V. For this purpose a MCP 1700 was used.

Ti Ar C’hazh (home base) design The Tracker itself did not have any way of accessing the API server without the use of a Wifi Chip, which would have greatly expanded the complexity of the design all while greatly increasing its power consumption. To avoid this, the home base was designed to be a relay for incoming GPS data. It was comprised of two main components: ● RAK3172 LoRA Transceiver Module ● Raspberry Pi Pico W 2 It used a RAK3172 to receive incoming transmissions sent by the Tracker over the LoRaP2P network. Then it would reformat the data into a JSON packet which could be sent to the API server using a Raspberry Pi pico 2 W’s Wifi chip. Figure 4. Example JSON packet showing the data parsed from the received LoRa transmission along with other pertinent information. The pico 2 W also provided an easy framework for automating the process of receiving GPS information, reformatting it and sending it out to an external server. Its role as a relay meant that it could both serve as a point of reference for the tracking while also being free from the tracker and its power restrictions. In other words, the home base could be connected to a wall outlet, via USB and a AC to DC converting brick, and connected to the local wifi. In order to reduce the size of the home base, a PCB design was drawn and printed out. It featured a RAK3172 chip connected via UART to a Raspberry Pi pico 2 W, as well as to a female SMA connector via the free RF port. The Raspberry Pi pico 2 W could be powered using the USB mirco-b port while an antenna could be attached to the homebase using the SMA port.

Server/API design Scope and Responsibilities The server is designed with a few objectives in mind. The first to create a centralized arbiter that stops the web application from receiving conflicting or incorrect information. If there was no

Another objective is to reduce the data load on the application so it can run quickly and smoothly. Alternative Designs The main point in the data flow that had multiple proposed designs was transmitting the data from the tracker to the server. The chosen solution uses LoRa Radio to send the GPS data from the tracker to the home base device in the house, which then uses the local internet connection to

relay the ping to the server. The solutions below are other ideas that were explored and why they were decided against. Cellular: The cellular network solution would involve a sim card located on the cat tracker itself. It would use the cellular network to upload GPS pings to the server. This would be accomplished by putting a cellular network transceiver on the tracker and whenever the tracker was within range of a tower the data would be sent to the server. The data flow of this proposed solution is seen below. Figure 19: Proposed Cellular network data flow On the surface, this solution can be seen as simpler and more efficient than what was decided. There are two main concerns that made this design unacceptable for the final product. The first concern is cellular network coverage. Outdoor cats may wander into wilderness or areas with a weak cellular connection. Many rural households already may not have cellular coverage at their house let alone their local forests or trails that the cat may venture into. This coverage issue would mean the cat could be close to home and not be locatable, this is unacceptable for a general use product.

===== PAGE 21 =====
Figure 20: Bell mobile coverage in northern Ontario For example, as seen in the figure above, many small towns in northern Ontario are fully without cellular coverage. Many areas that do have coverage have weak coverage, these weaker cellular networks may be easily lost by a cat wandering, especially somewhere like a forest that weakens signals. The next concern with the cellular solution is that the cellular network is a metered connection, users would have to pay to send the GPS pings over cellular. As a product this would likely require a monthly subscription that covers the data usage opposed to a one time purchase. This would deter users from using the tracker but some users would find themselves in the situation where they bought the hardware and eventually cancel the subscription. These users would have tracking hardware that is unusable to them unless they benign their subscription again. The chosen solution avoids both of these pitfalls. The home base radio receiver makes the range centered around the house so that someone in the suburbs of Toronto and someone in rural northern Ontario have the same experience. Then the home base uses the local WiFi to send the data to the server, avoiding the metered connection of the cellular network. Upload at House: The second proposed solution was to forgo live tracking and upload the cat's path when it returns home. This would avoid metered connection and poor cellular coverage but the user would not be able to find the cat if it was missing. This would require the user to look at historical data and see where their cat frequents then search for it based on that data. This solution is better for those interested in what their pet gets up to outdoors and is more of a novelty than a useful tool. The implemented solution avoids this by using LoRa local radio transmission to send the data to the house and uses the house's internet connection to upload to the server. Live tracking was determined to be a fundamental part of the product that could not be forgone. Measurement and Testing Procedures Hardware Verification Procedure The Tracker was initially tested by connecting it via UART using a USB-to-UART converter with a laptop running Arduino IDE in order to see whether the RAK3172 could

===== PAGE 23 =====
interface with the Quetel LR80 GPS and if the GPS gave the correct output. The RAK3172 was also programmed to show, in the command output of the Arduino IDE, the data that it had formatted to send through the LoRaP2P network. Meanwhile, the home base had more components to test. First, the Raspberry Pi needed to be able to interface with a laptop through USB using rshell. Next, the WiFi chip needed to work properly. Error messages relating to the Wifi chip meant that it was not able to connect to any Wifi network and thus the firmware on the pico 2 W needed to be changed. Once the Wifi chip was not showing errors, the communication between the pico 2 W and the RAK3172 was checked by running the main script of the home base. Everytime the home base main script ran, it did an initialization sequence where the first message the pico 2 W sent was “AT”. If the RAK3172 responded with “OK” (shown in the stdout) then communication worked. Otherwise, the firmware of the pico 2 W likely needed to be changed. To test if the pico 2 W sent the data correctly, the Tracker needed to be set up in advance and already started sending GPS data every 10 seconds. Everytime the pico 2 W received a signal, it would parse the data, print out to stdout and attempt to send it to the API server. If everything worked, then the pico 2 W would have printed a 201 HTTP response to stdout, and the endpoint for the server would have shown the received JSON on its webpage. Hardware Testing Procedure The Tracker and Home base always needed to be tested together, especially when they were only connected to power. The only way that a message could be successfully received was if the tracker had gathered the GPS coordinates and successfully sent it through the LoRaP2P network, and if the Home base had successfully received and parsed the LoRa message while relaying it to the API server. The pico 2W on the home base had its onboard LED flash whenever a JSON “ping” was being sent to the API server and thus was a very good indicator of the tracker having successfully sent a GPS ping and the home having successfully parsed it. Testing the range of the tracker was the only interesting metric that was needed for testing. The tracker and the home base both needed to be powered and a hotspot( the hotspot’s authentication credentials needed to be hardcoded into the script) needed to be set up as well. Next, the person with the tracker needed to venture outside where the GPS could have an unobstructed view of the sky and then start moving away from the homebase. The coordinates received by the tracker on the API endpoint could either be plugged into google maps, or, if the app made by Minh was being used, then the location could be directly assessed on the app’s live map view and dashboard. After the tests were done, the JSON “pings” could also be collected from the API endpoint’s webpage after the fact and plotted onto a map to assess details of when and where the pings cutout and what path the tester had taken. From this procedure, a maximum range could be meaningfully measured and documented, taking into account the environment that the tests were being held in.
